# Guardrails

Правила для coding-агентов в этом репозитории.
Это не style guide — это контракт безопасности.

---

## 1. Незыблемые правила

Без исключений.

1. **Не выдумывать архитектуру.** Не проектировать новые package layout, namespace-структуры или межпакетные зависимости без явного подтверждения от пользователя.
2. **Не ломать существующие слои.** Иерархия `endpoint → controller → repository` (CRUD) и `endpoint → controller → use case → repository` (DDD) зафиксирована. Бизнес-логика не принадлежит в роутер или репозиторий.
3. **Не добавлять `__init__.py` в namespace package директории.** `src/axiom/`, `src/axiom/oltp/`, `src/axiom/olap/` — PEP 420 implicit namespace packages; они должны оставаться без `__init__.py`.
4. **Не бросать сырой `Exception`.** Все исключения должны наследовать от `BaseError` (`axiom.core.exceptions.base`). Исключения сторонних библиотек оборачивать в `Axiom{Name}Error` на boundary.
5. **Не отгружать непокрытый тестами код.** Каждая новая функция, метод или класс требует хотя бы одного теста. Stub-only работа должна быть зафиксирована в `CONCERNS.md`.
6. **`make test` и `make check-precommit` должны проходить** перед тем как считать задачу выполненной.

## 2. Опасные паттерны

| Поведение | Почему опасно |
|---|---|
| Добавить `# type: ignore` или `# noqa` без комментария | Скрывает реальные ошибки; и так слишком много — не усугублять |
| Смешивать loguru и structlog | Это уже tech debt проекта; канонический логгер — `axiom.core.logger.get_logger()` |
| Создавать stub-only пакет без записи в `CONCERNS.md` | Потребители получают молчаливый no-op; записать статус `stub-only` с описанием impact |
| Менять `BaseError`, `AsyncCacheBackend`, `BaseController` или другие базовые классы | Может тихо сломать все downstream-пакеты; требует явного подтверждения пользователя |
| Использовать `dill.loads` с данными из ненадёжных источников | Произвольное выполнение кода; `DillStrategy` намеренно изолирован — не менять |
| Помещать доменную логику в `router`, `schema` или `repository` | Размазывает бизнес-правила по неправильным слоям; логика живёт в domain entities и use cases |
| Переименовывать или перемещать пакеты/namespace | Ломает всех потребителей; требует ADR в `docs/decisions.md` |
| Добавлять абстракцию для одного use case | Преждевременная абстракция; добавлять когда появится второй потребитель |
| Глобально расширять `disable_error_code` в `mypy.ini` | Уже слишком широко; добавлять только точечные per-file подавления |

## 3. Качество кода

- **Именование:**
  - Async-варианты: класс `AsyncFooBackend`, файл `async_.py`
  - Sync-варианты: класс `SyncFooBackend`, файл `sync.py`
  - `exception/` — всегда sub-package (директория + `__init__.py`), никогда плоский `.py`
  - `__all__` сортируется алфавитно
- **Каждый исходный файл** обязан иметь module docstring: `"""axiom.{full.namespace}.{module} — описание."""`
- **Полные type annotations** на всех публичных функциях и методах
- **Без path aliases** — только полные dotted-импорты: `from axiom.cache.base import AsyncCacheBackend`
- **`async_.py`/`sync.py` паритет** — если есть async-вариант, должен быть и sync (или явный ADR об исключении)
- **Никаких преждевременных абстракций** — три похожих строки лучше, чем ранний helper

## 4. Тесты

- **Unit-тесты** — обязательны для всей доменной логики; без DB, без I/O, без внешних вызовов
- **Integration-тесты** — нужны, когда нельзя уверенно проверить сценарий без реального сервиса; сначала fakes (`fakeredis`, `aiosqlite`), testcontainers — когда нужна production-like точность
- **Не мокать клиентские классы** — использовать fakes или in-memory эквиваленты; мок, который проходит пока прод ломается, хуже отсутствия тестов
- **Edge cases, которые нужно покрыть всегда**: `None`-значения, пустые коллекции, неверные типы, failure/exception пути
- **`asyncio_mode = "auto"`** должен быть выставлен — `@pytest.mark.asyncio` не нужен
- **Паттерн тестовых данных**: `_make_entity(**kwargs)` или `_order(**kwargs)` — дефолты с переопределением; не fixture для простых данных
- **Stub-only пакеты** должны иметь пустой `tests/__init__.py` И запись в `CONCERNS.md` — не писать тесты, которые проходят ничего не делая

## 5. Документация

После любого значимого изменения — проверить и обновить нужные файлы.
Полный checklist: [`docs/checklists/major-change.md`](docs/checklists/major-change.md)

**Минимально необходимые обновления по типу изменения:**

| Изменение | Файлы для обновления |
|---|---|
| Новый пакет | `STRUCTURE.md`, `ARCHITECTURE.md`, `docs/plugins.md`, `README.md` |
| Новый backend | `INTEGRATIONS.md`, `{package}/README.md` |
| Новый паттерн | `docs/patterns.md`, соответствующие `examples/*.md` |
| Архитектурное решение | `docs/decisions.md` (новая запись D-NNN) |
| Изменение публичного API | `{package}/README.md`, `docs/plugins.md` |
| Новый tech debt | `CONCERNS.md` |
| Tech debt устранён | Удалить или обновить запись в `CONCERNS.md` |

**Нельзя писать документацию, описывающую код, которого ещё не существует.**
**Нельзя пропускать обновление документации при изменении публичного API.**

## 6. Управление изменениями

- **Checklist нового плагина** — [`examples/plugin/README.md`](examples/plugin/README.md); следовать точно
- **Stub-only работа**: допустима как scaffolding; обязательно записать в `CONCERNS.md` со статусом `stub-only` и описанием impact
- **Breaking changes** базового класса или публичного API требуют:
  1. Найти всех dependents прежде чем трогать что-либо
  2. Запись ADR в `docs/decisions.md`
  3. Обновить все затронутые тесты
- **Не добавлять пакеты в workspace молча** — обновить `[tool.uv.workspace] members` в корневом `pyproject.toml`
- **Не добавлять пути в `mypy_path` в `mypy.ini`** если директория ещё не существует (известная проблема с `axiom-serialization`)

---

## 7. Когда остановиться и задать вопрос

Остановиться и спросить пользователя перед продолжением, если:

- Требования неясны или неоднозначны
- Задача затрагивает базовый класс (`BaseError`, `BaseDomainDC`, `AsyncBaseController`, любой ABC)
- Задача требует изменения package layout или namespace-структуры
- Задача удаляет или переименовывает публичный API
- Задача вводит новый архитектурный паттерн, не задокументированный в `docs/patterns.md`
- Нет чёткого reference-примера в `examples/` или `docs/patterns.md`
- Дизайн требует выбора между двумя вариантами с неочевидными последствиями
- Нужны reference repositories из внешних проектов, чтобы понять правильный паттерн

**Явно запрашивать** reference repos или canonical examples если паттерн реализации неясен.
**Не продолжать наугад** в решениях, затрагивающих архитектуру, публичный API или test strategy.

## Топ-10 правил для `AGENTS.md`

Вставить в `AGENTS.md` для быстрого онбординга:

1. Не выдумывать архитектуру и контракты — сначала читать код
2. Исключения наследуют от `BaseError`; сторонние ошибки оборачивать в `Axiom{Name}Error` на boundary
3. Никакого `__init__.py` в `src/axiom/`, `src/axiom/oltp/`, `src/axiom/olap/`
4. Бизнес-логика живёт в domain entities и use cases — не в роутерах, схемах и репозиториях
5. Unit-тесты для логики; integration-тесты когда нужен реальный сервис; fakes вместо моков
6. `make test` и `make check-precommit` должны проходить перед завершением задачи
7. Каждый новый файл требует module docstring с полным namespace: `"""axiom.x.y.z — описание."""`
8. Использовать `axiom.core.logger.get_logger()` — не добавлять structlog в новый код
9. Stub-only работа фиксируется в `CONCERNS.md`; никогда не отмечать как завершённую без реализации
10. Остановиться и спросить при касании базовых классов, namespace или публичного API
